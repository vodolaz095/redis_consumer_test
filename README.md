Redis Consumer Test
===============================
Попытка разобраться, можно ли использовать PubSub очереди в редис как надёжный механизм доставки.

Для случая, когда
1. сообщения публикуются раз в 1 мс
2. сообщения обрабатываются 100 мс
3. используется 10 потребителей

Более 87% сообщений теряются!

```

2025/12/06 23:57:02 Приложение остановлено!
2025/12/06 23:57:02 Интервал отправки: 1ms
2025/12/06 23:57:02 Задержка обработки: 100ms
2025/12/06 23:57:02 Количество потребителей : 10
2025/12/06 23:57:02 Статистика: отправлено 1024, принято 136, доля потерянных сообщений: 87.00%

```

Вывод
================================
В случае частых публикаций и долгой работы потребителя много сообщений потеряется.

Redis Pub/Sub НЕ ЯВЛЯЕТСЯ надёжным транспортом сообщений и не реализует надёжную очередь доставки.

Буферизированный канал Go надёжную очередь реализует.

Указанный в https://github.com/vodolaz095/grq?tab=readme-ov-file#protocol-definition протокол её реализует.

RabbitMQ её реализует.

Как проверить самому
================================

1. Запустить redis - можно через docker/podman compose
    ```shell
    $ make podman/up
    $ make docker/up
    ```

2. Изменить константы в `main.go`
3. Запустить по `make start`
4. CTRL+C - останавливает приложение и выводит статистику
